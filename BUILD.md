# BUILD.md — как добавлять фичи в MRPG Overwatch без багов

> **Роль этого файла.** Он отвечает на один вопрос: **«я добавляю X — что я обязан
> сделать, чтобы не оставить баг?»** Это **процедура**: чек-листы по типу задачи.
> Он короткий специально. Сюда стоит писать: рецепты и обязательные шаги при
> добавлении контента/инфраструктуры.
> **Не сюда:** объяснение языка и движка (→ `CODESTYLE.md`), описание того, какие
> системы уже есть в коде (→ `ARCHITECTURE.md`), порядок чтения файлов (→ `.agents/AGENTS.md`).

## Порядок работы (всегда)

1. Прочитай этот файл (`BUILD.md`).
2. Прочитай `ARCHITECTURE.md` — карту того, что уже реализовано. Если делаешь
   фичу, похожую на существующую, открой её файл и повтори паттерн.
3. Найди в разделе «Рецепты» ниже свой тип задачи и выполни чек-лист целиком.
4. Если рецепта нет — см. «Новая инфраструктура» внизу.
5. **В конце ОБЯЗАТЕЛЬНО запусти шлагбаум:**
   ```
   python tools/validate.py
   ```
   Если он не печатает `0 error(s)` — ты что-то забыл. Чини по адресу `файл:строка`
   из вывода. Не сдавай задачу с красным валидатором.
6. Обнови `ARCHITECTURE.md`, если добавил/изменил систему (см. последний раздел).

## Золотые правила (нарушение = баг через полгода)

1. **Никогда не пиши `.active_buff_*.append(...)` или `.hud_buff_*.append(...)`
   вручную.** Используй макросы `trackBuff` / `applyVisibleDebuff` (см. рецепт A).
   Валидатор ловит ручные обходы (проверка D2).
2. **Все глобальные сущности (Effects, Texts) должны отслеживаться.**
   Запрещены сырые вызовы `createEffect`, `createInWorldText` без последующего 
   сохранения ID через `applyVisibleDebuff`, `trackBuff` или `trackPersistentEntity`.
   Валидатор запретит сборку при утечках памяти (проверка G).
2. **Любой видимый эффект на игрока обязан иметь HUD-надпись.** Способ, который
   это гарантирует, — `applyVisibleDebuff` (механика + HUD одним вызовом).
3. **Стат-эффект (SPEED/DAMAGE/HEALTH/DAMAGE_RECEIVED/HEALING_RECEIVED/
   PROJECTILE_SPEED) обязан передавать И `action`, И `value`.** Иначе при истечении
   стат «восстановится на 0» — тихая порча стата навсегда. Валидатор это ловит
   (проверка E, type-aware).
4. **Никаких магических чисел для индексов массивов** — только enum из
   `array_schemas.opy`. Длина инициализатора массива обязана совпадать с числом
   полей enum (валидатор, проверка B).
5. **Каждый цикл `while`/`for` содержит `wait()`** (см. `CODESTYLE.md §1.4`).
6. **Незаконченную заготовку помечай `# stub`** в объявлении (валидатор не
   будет считать её ошибкой, но напомнит WARN'ом).

## Хребет проекта (через что всё течёт)

Это две системы, которые знают про эффекты. Всё остальное должно пользоваться ими,
а не изобретать своё.

- **`systems/effects_lifecycle.opy`** — хранит, продлевает и **автоматически
  удаляет** все временные эффекты (визуал, изменения статов, health pool, heal-
  over-time, флаги). Удаление и откат стата происходят сами, когда истекает время.
- **HUD-баффы** (правый список) — `hud_buff_*` массивы, рисуются в `core/hud.opy`
  (5 слотов, лимит осознанный — экономия 128 текстов).

Регистрируешь эффект → он сам уберётся и сам откатит стат. **Но только если ты
зарегистрировал его правильно** (рецепты ниже).

- **Мусорная корзина (Garbage Collection)** — `core/player_leave.opy`.
  Хранит ID постоянных эффектов (`gc_ids`, `gc_types`), которые не являются временными баффами (например, текст статуса). Они автоматически уничтожаются при выходе игрока из матча. Регистрируй их через макрос `trackPersistentEntity`.

## Рецепты

### A. Добавить видимый бафф/дебафф на игрока (или бота, или victim)

Самый частый случай. Один вызов делает всё:

```python
# СТАТ-эффект (нужны action + value):
victim.applyVisibleDebuff(BuffId.MY_STAT, victim, getTotalTimeElapsed() + duration,
    EffectType.DAMAGE_RECEIVED, EffectAction.MODIFY_STAT, bonusValue,
    "{0} My Debuff".format(iconString(Icon.SKULL)), Color.PURPLE)

# ВИЗУАЛЬНЫЙ эффект-сущность (action/value не нужны -> 0, 0):
createEffect(getAllPlayers(), Effect.X, Color.PURPLE, victim, 1, EffectReeval.VISIBILITY_POSITION_AND_RADIUS)
victim.applyVisibleDebuff(BuffId.MY_VISUAL, getLastCreatedEntity(),
    getTotalTimeElapsed() + duration, EffectType.ENTITY, 0, 0,
    "{0} My Effect".format(iconString(Icon.X)), Color.PURPLE)
```

Чек-лист:
- [ ] Добавил `BuffId.MY_X` в enum `BuffId` (`settings/constants.opy`). Визуал и
      стат — **разные** `BuffId` (правило Split IDs, рецепт B).
- [ ] Вызвал `applyVisibleDebuff(...)` (видимый) или `trackBuff(...)` (если эффект
      намеренно невидим — редко).
- [ ] Для **нового** `EffectType` стата — см. рецепт C (иначе стат не откатится!).
- [ ] Не пишу `.append` руками.
- [ ] `python tools/validate.py` → `0 error(s)`.

Эффект без HUD (например, иммунитет на спавне) — это **состояние зоны**, а не
бафф. Не регистрируй его в lifecycle, управляй обычной переменной.

### B. Продлить / обновить существующий эффект

Не плоди дубликаты в массивах. Обнови по индексу (атомарно):

```python
if BuffId.MY_X in victim.active_buff_ids:
    victim.active_buff_expire_times[victim.active_buff_ids.index(BuffId.MY_X)] = newExpire
if BuffId.MY_X in victim.hud_buff_ids:
    victim.hud_buff_expires[victim.hud_buff_ids.index(BuffId.MY_X)] = newExpire
```

Для пачки ID разом — `extend_buff()` (формат `[[ids...], newExpire]`).

**Жёсткие правила продления (нарушение = баг или падение сервера):**

1. **Атомарность вместо циклов.** СТРОГО ЗАПРЕЩЕНО использовать локальные циклы
   `while` + `wait(0.016)` для продления таймеров баффов при динамических событиях
   (например, при каждом тике урона). Обновление таймера/значения должно быть
   мгновенным — прямым поиском по индексу, как выше. (Почему циклы запрещены вообще —
   `CODESTYLE §1.4` и `§6.1`.)
2. **Агрегация стаков.** Если бафф стакается (как Cursed Mark), запрещено плодить новый
   элемент в массивах `active_buff_*` на каждый стак. Бафф регистрируется **один раз**
   (на 1-м стаке через рецепт A). При последующих стаках обновляй значение слота:
   ```python
   victim.active_buff_values[victim.active_buff_ids.index(BuffId.MY_STAT)] += bonus
   ```
   **Паттерн «N singletons + стаки»:** (Эталон: `classes/reader/prayer_of_salvation.opy` и `prayer_of_courage.opy`). Если способность вешает несколько баффов (визуал, активный флаг) и один стакающийся, ВСЕГДА сначала безусловно продлевай все баффы (`extendBuff` или `extendBuffAll` для дублирующихся ID), затем на 1-м стаке создавай singletons, и только затем обрабатывай логику стаков.
3. **Дублируемые эффекты (Stacking).** Если бафф НАМЕРЕННО является дублируемым (может иметь несколько независимых копий в `active_buff_ids`):
   - **ОБЯЗАТЕЛЬНО** добавь маркер `# DUPLICATABLE` напротив его `BuffId` в `constants.opy`.
   - **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** использовать макрос `extendBuff` (он сломается из-за `.index()`). Используй только `extendBuffAll`! Валидатор проверяет это (правило K).
4. **Per-Attacker эффекты (Stacking от разных игроков).** Если эффект может быть наложен на одну цель **разными** игроками независимо (каждый со своим таймером и стаками, как Cursed Mark):
   - ИСПОЛЬЗУЙ **только** API `trackAttackerBuff`, `applyVisibleAttackerDebuff` и `extendAttackerBuff`.
   - Это гарантирует, что система не смешает баффы от двух игроков.
   - Это защищает от багов `.index()`, так как поиск ведется по комбинации `(BuffId + Source)`.
5. **Split IDs.** Если способность вешает И визуал, И стат — у них ОБЯЗАНЫ быть разные
   `BuffId` (рецепт A, золотое правило): иначе `.index()` найдёт не тот слот.
6. **Перманентные эффекты (Immune to Death).** Если эффект НЕ ДОЛЖЕН пропадать после смерти цели (например, проклятие на 10 минут, которое висит даже после респауна):
   - ИСПОЛЬЗУЙ стандартный макрос `trackBuff` или `applyVisibleDebuff`.
   - **Сразу следующей строкой** вызови модификатор `victim.setLastVisibleFlag(BuffFlag.PERSIST_ON_DEATH)`.
   - Система `respawn.opy` не обнулит таймер благодаря этому флагу. Вызов модификатора сразу после трекинга гарантированно потокобезопасен.

### C. Добавить НОВЫЙ стат (растёт с уровнем, баффается, имеет лимит)

Новая компонентная архитектура (Base + Modifiers) делает это намного проще и безопаснее. Пройди пункты:

- [ ] Добавь поле в конец enum `EffectType` (`settings/constants.opy`). **Важно:** значение в `EffectType` используется как индекс в массивах `base_stats` и `mod_stats`.
- [ ] Добавь макросы `base_stat_X` и `mod_stat_X` в `array_schemas.opy` с индексом нового `EffectType`.
- [ ] Добавь обновление нативного значения в макрос `apply_all_stats()` (например, `self.setMoveSpeed(max(0, self.base_stat_speed + self.mod_stat_speed))`).
- [ ] Увеличь размер инициализатора массивов `base_stats` и `mod_stats` в `core/init.opy` **во ВСЕХ блоках init** (`Player Init` И `Init Dummy Bot`), добавив новый стартовый элемент (например, 100 для base, 0 для mod). Забыл бота — баг только на ботах. Валидатор ловит это.
- [ ] Лимит: поле в enum `StatLimit` + макрос `statLimitX` (`array_schemas.opy`), обновление массива `stats_limit = [...]` в `init.opy`.
- [ ] Если стат перманентно меняется (например, прокачка за уровень), изменяй **только** `base_stat_X` и вызывай `apply_all_stats()`.
- [ ] Если стат баффается на время, просто используй `trackBuff`/`applyVisibleDebuff` с твоим `EffectType`. Система автоматически обновит `mod_stats` и вызовет `apply_all_stats()` при применении и истечении эффекта!
- [ ] Отображение в `core/hud.opy`, если стат показывается.

### D. Добавить способность

- [ ] Новый файл в `classes/<класс>/<способность>.opy`, первая строка
      `#!mainFile "../../main.opy"`.
- [ ] `#!include` нового файла в `main.opy` (в секции `--- Classes ---`).
- [ ] Переменные состояния способности — через схему в `array_schemas.opy`
      (enum + member-макросы), а не новые playervar по одному.
- [ ] Подпрограммы способности — объявить в `variables/subroutines.opy`.
- [ ] Эффекты способности — через рецепт A (`applyVisibleDebuff`).
- [ ] Сужай событие: `@Event eachPlayer` + ранние дешёвые `@Condition`
      (`CODESTYLE §6.3`).
- [ ] Очистка при смерти/смене класса, если способность оставляет состояние
      (см. рецепт E).
- [ ] `python tools/validate.py` → зелёно.

### E. Очистка состояния (смерть / выход / смена тира)

Любая система с «живущим» состоянием обязана убирать его. Сейчас в проекте это
**слабое место** (см. `ARCHITECTURE.md` → «Известные швы»). Если добавляешь
persistent-состояние:

- [ ] Смерть (`@Event playerDied` в `core/respawn.opy`): сбрось/очисти.
- [ ] Выход (`@Event playerLeft` в `core/player_leave.opy`):
      Механизм GC уже чистит все трекаемые эффекты (баффы и `gc_ids`). Если твоё
      состояние требует очистки помимо визуальных эффектов (сброс массивов,
      влияющих на глобальную логику) — добавляй сюда.
- [ ] Смена тира (`tier_up_reset`, пока заготовка): сбрось бонусы прошлого тира.

### F. Добавить зону

- [ ] Значение в enum `Zone` (`settings/constants.opy`).
- [ ] Координаты/радиус в `<зона>/coordinates.opy`.
- [ ] Вход/выход обновляют `eventPlayer.current_zone`.
- [ ] Создание/удаление HUD зоны парные (вошёл создал — вышел удалил).
- [ ] `#!include` всех файлов зоны в `main.opy`.

## Новая инфраструктура (НЕ контент, а новый механизм) — читай при отсутствии рецепта

Это ответ на главный риск: «защита рассчитана на старые системы, а я делаю
принципиально новое». Правило простое:

> **Сначала попытайся выразить задачу через существующий хребет**
> (`effects_lifecycle` + макросы). Новый примитив создавай ТОЛЬКО если иначе
> физически невозможно. Каждый лишний «хребет» — это новый набор швов.

Если новый примитив всё-таки необходим, ты в той же сессии **обязан принести для
него защиту** (иначе следующая нейросеть про него забудет):

1. **Добавь проверку инварианта в `tools/validate.py`.** Если у твоей системы есть
   правило вида «при X всегда должно быть Y» (что-то регистрируется — должно
   чиститься; добавил в enum — должен обработать везде) — напиши функцию-чек и
   зарегистрируй её в списке `CHECKS`. Шаблон: см. любую `check_*` функцию;
   добавить чек = одна функция + одна строка в `CHECKS`. Кривой чек лучше, чем
   никакого.
2. **Добавь рецепт в `BUILD.md`** (этот файл) — короткий чек-лист «как добавлять
   контент поверх твоего примитива».
3. **Добавь систему в `ARCHITECTURE.md`** — что делает, какие обязательства по
   очистке, через какой макрос/подпрограмму с ней работать.

Так защита растёт вместе с проектом, а не отстаёт от него.

## Когда `validate.py` красный

- **`branch_data содержит N, ожидается M`** — ты изменил enum, но не обновил
  инициализатор массива (или наоборот). Синхронизируй длину.
- **`subroutine X объявлена, но def не найден`** — либо напиши `def X():`, либо,
  если это заготовка, пометь объявление `# stub`.
- **`buff_append_args для стат-типа содержит < 6`** — добавь `action` и `value`.
- **`slot занят N раз`** — два `playervar`/`globalvar`/`subroutine` на один номер
  слота. Дай свободный.
- **`ручной .append вне effects_lifecycle` (WARN)** — перепиши на `trackBuff` /
  `applyVisibleDebuff`.
- **`Сырой вызов X не отслеживается`** — используй `trackPersistentEntity`, 
  `trackBuff` или `applyVisibleDebuff` после создания эффекта, чтобы GC смог 
  убрать его при выходе игрока.

Полное описание каждой проверки — в шапке `tools/validate.py`.
