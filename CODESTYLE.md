# Предписание по написанию OverPy кода

Данный документ определяет правила написания кода для проекта MRPG Overwatch.
Предназначен для разработчиков и нейросетей.

---

## 1. Базовые конструкции OverPy

### 1.1 Структура правил (rules)

```python
rule "Название правила на английском":
    @Event eachPlayer          # или global
    @Condition условие1
    @Condition условие2        # условия объединяются через AND
    
    # тело правила
    действие1
    действие2
```

- `@Event eachPlayer` — правило выполняется для каждого игрока, доступен `eventPlayer`
- `@Event global` — глобальное правило, выполняется один раз
- `@Condition` — правило срабатывает только когда ВСЕ условия истинны
- Все `@Condition` должны идти до тела правила
- `@Disabled` — отключает правило (для отладки)

### 1.2 Переменные

```python
# Объявление (в файлах variables/*.opy)
playervar имя_переменной слот_номер    # переменная игрока (макс 128)
globalvar имя_переменной слот_номер    # глобальная переменная (макс 128)

# Использование
eventPlayer.имя_переменной = значение  # для playervar
имя_переменной = значение              # для globalvar
```

- Слот — число от 0 до 127, каждый слот уникален
- Имя переменной — snake_case
- Максимальный размер массива — 1000 элементов

### 1.3 Подпрограммы (subroutines)

```python
# Объявление (в variables/subroutines.opy)
subroutine имя_подпрограммы

# Определение
def имя_подпрограммы():
    @Name "Читаемое название"
    # тело

# Вызов
имя_подпрограммы()
```

- Подпрограммы НЕ поддерживают аргументы напрямую
- Для передачи данных — записывай во временные переменные перед вызовом
- `def` в OverPy компилируется в subroutine Workshop, а НЕ в обычную функцию

### 1.4 Циклы и условия

```python
# for
for переменная in range(число):
    действие
    wait(0.016)    # ОБЯЗАТЕЛЬНО wait в циклах!

# while  
while условие:
    действие
    wait(0.016)

# if/elif/else
if условие:
    действие
elif другое_условие:
    действие
else:
    действие

# loop (повтор правила при сохранении условий)
if ruleCondition:
    loop()
```

> **ВАЖНО**: Каждый цикл ДОЛЖЕН содержать `wait()`, иначе сервер зависнет.
> Минимальный wait = 0.016 секунд.
> Классические циклы используются только для растянутых во времени процессов или таймеров. Для перебора элементов массива всегда используйте мапперы (см. 6.1).

### 1.5 Константы компиляции

```python
#!define ИМЯ_КОНСТАНТЫ значение

# Пример
#!define SPAWN_BUBBLE_RADIUS 45.3
```

- Простая текстовая подстановка при компиляции
- Не создаёт переменную Workshop — не занимает слот
- Используй для фиксированных числовых значений

### 1.6 Включение файлов

```python
#!include "путь/к/файлу.opy"
```

- Каждый включаемый файл должен начинаться с `#!mainFile "путь/к/main.opy"`
- Порядок включения важен — зависимости должны быть подключены раньше

### 1.7 Полезные функции Workshop

```python
# Таймер / ожидание
wait(секунды)

# Эффекты
createEffect(visibility, тип, цвет, позиция, радиус, reeval)
destroyEffect(id)
playEffect(visibility, тип, цвет, позиция, радиус)  # мгновенный

# HUD
hudSubheaderText(видимость, текст, позиция, сортировка, цвет, reeval)
progressBarHud(видимость, процент, текст, позиция, сортировка, цвет1, цвет2, reeval)

# Игрок
eventPlayer.getPosition()
eventPlayer.getEyePosition()
eventPlayer.isAlive()
eventPlayer.isMoving()
eventPlayer.isHoldingButton(Button.XXX)
eventPlayer.setMoveSpeed(процент)
eventPlayer.setDamageDealt(процент)
eventPlayer.setMaxHealth(процент)

# Массивы
array.append(элемент)
del array[индекс]
len(array)
array.filter(lambda x: условие)
array.map(lambda x: выражение)
array.index(значение)  # возвращает -1 если не найден

# Утилиты
getTotalTimeElapsed()     # время с начала матча
getLastCreatedEntity()    # ID последнего созданного эффекта
getLastCreatedText()      # ID последнего созданного текста
evalOnce(выражение)       # вычислить один раз и запомнить
```

## 2. Продвинутые конструкции: управление переменными

### 2.1 Проблема

Workshop ограничен 128 переменными игрока. Нет объектов, словарей, структур.
Сложные данные приходится хранить в массивах. Код вида `array[5][2]` нечитаем.

### 2.2 Многомерные массивы и вложенность

Движок поддерживает многомерные и "рваные" (ragged) массивы. Функции `len()`, `filter()` и другие работают на любой глубине. При уровне вложенности 1 и больше необходимо придерживаться специфических правил:

- **Приоритет одномерности**: Если для задачи возможно использовать одномерный массив (например, параллельные одномерные массивы) без значительной потери ресурсов, то предпочтительнее использовать одномерный массив.
- **Тестирование**: Все операции с массивами, которые не были протестированы, могут не работать. Перед применением они нуждаются в тестировании.
- **Сплющивание `append`**: Метод `.append(array)` распаковывает первый уровень добавляемого массива. Для сохранения массива как цельного элемента используй двойные скобки: `.append([[array]])`.
- **Лимит компилятора OverPy**: Прямое присвоение поддерживается максимум до 3-х уровней (`arr[0][0][0] = X`). Запись на глубину 4+ вызовет ошибку `Cannot assign to 4d array`. Для изменения переписывай структуру на 3-м уровне: `arr[0][0][0] = [[X]]`.
- **Удаление**: Команда `del array[index]` работает исключительно с корневым (первым) уровнем. Прямое удаление элементов из внутренних массивов невозможно.
- **Бесшумные ошибки**: Запрос индекса за пределами массива или запрос индекса у числа (например, `15[2]`) возвращает `0` и не прерывает выполнение.

### 2.3 Решение: Enum-индексы

Определяй `enum` для каждого массива-структуры:

```python
enum PrayerSalvation:
    PROGRESS = 0,
    HUD_IDS,
    CAST_DURATION,
    CAN_CAST_MOVING,
    IS_ACTIVE

# Использование:
eventPlayer.reader_prayer_of_salvation[PrayerSalvation.PROGRESS]++
eventPlayer.reader_prayer_of_salvation[PrayerSalvation.IS_ACTIVE] = true
```

- Enum компилируется в число — нулевой оверхед в Workshop
- Каждый массив-структура должен иметь свой enum
- Описание полей — в комментарии над enum

### 2.4 Решение: Member-макросы (сокращённый доступ)

```python
macro Player.prayerProgress = self.reader_prayer_of_salvation[PrayerSalvation.PROGRESS]
macro Player.prayerIsActive = self.reader_prayer_of_salvation[PrayerSalvation.IS_ACTIVE]

# Использование (компилируется в тот же код что и enum-индексы):
eventPlayer.prayerProgress++
eventPlayer.prayerIsActive = true
```

- `macro Player.xxx` создаёт свойство, доступное через `player.xxx`
- `self` заменяется на конкретного игрока при компиляции
- Нулевой оверхед — это текстовая подстановка
- Определяй в `settings/array_schemas.opy`

### 2.5 Решение: Функциональные макросы

```python
macro add(a, b): a + b
macro Player.setPowerLevel(level):
    self.setMaxHealth(level * 300)
    self.setDamageDealt(100 + level * 20)
```

- Для повторяющихся вычислений
- Макросы НЕ создают subroutine — код вставляется inline

### 2.6 Файл array_schemas.opy

Все enum-схемы и member-макросы для массивов хранятся в `settings/array_schemas.opy`.

Структура файла:
```python
# ============================================================
# СХЕМЫ МАССИВОВ — [ИМЯ МАССИВА]
# ============================================================
# описание массива

enum ИмяСхемы:
    ПОЛЕ_1 = 0,
    ПОЛЕ_2,
    ПОЛЕ_3

# Member-макросы
macro Player.кратокеИмя = self.имя_переменной[ИмяСхемы.ПОЛЕ_1]
```

**Правила:**
- При добавлении нового поля — добавь в enum и напиши комментарий
- Неиспользуемые слоты помечай как `_UNUSED_N`
- Member-макросы именуй в camelCase: `prayerProgress`, `statLimitHealth`

## 3. Стиль кода

### 3.1 Именование

| Что | Стиль | Пример |
|-----|-------|--------|
| Переменные (playervar/globalvar) | snake_case | `stat_damage`, `in_blood_shop` |
| Enum типы | PascalCase | `PrayerSalvation`, `EffectEntry` |
| Enum значения | UPPER_SNAKE_CASE | `CAST_DURATION`, `IS_ACTIVE` |



- Комментируй ЗАЧЕМ, а не ЧТО
- Каждый нетривиальный блок кода должен иметь комментарий
- Не удаляй существующие комментарии при рефакторинге

## 4. Глобальные системы инфраструктуры

Проект использует универсальные глобальные системы для обработки общих задач. Прямое (локальное) создание и ручное удаление эффектов без необходимости запрещено — всегда используйте централизованную инфраструктуру.

### 4.1 Система жизненного цикла эффектов (effects_lifecycle.opy)

Управляет хранением, продлением и автоматическим удалением баффов, дебаффов, визуальных эффектов и временных изменений статов.
- **Добавление эффекта**: Данные передаются через массив `eventPlayer.buff_append_args`, затем вызывается `add_buff()`.
- **Продление эффекта**: Для существующих эффектов (статов/сущностей) вызывается `extend_buff()` с массивом ID и новым временем истечения.
- **Удаление эффекта**: Происходит автоматически в правиле `Process player buffs`, когда `getTotalTimeElapsed()` превышает заданное время истечения. Система сама уничтожает Entity, восстанавливает статы и очищает массивы.

**Жесткие правила по работе с эффектами:**
1. **Разделение ID (Split IDs)**: Если способность вешает на игрока и визуальный эффект, и изменение статов, они ОБЯЗАНЫ иметь разные `BuffId` (например, `ABILITY_VISUAL` и `ABILITY_STAT`). Это необходимо для точного и быстрого поиска нужного баффа через `.index()`.
2. **Атомарность вместо циклов (No Loops)**: СТРОГО ЗАПРЕЩЕНО использовать локальные циклы `while` и `wait(0.016)` для продления таймеров баффов при динамических событиях (например, при получении урона). Обновление таймера или значения должно происходить мгновенно (атомарно) через прямой поиск по индексу: `victim.active_buff_expire_times[victim.active_buff_ids.index(BuffId.ABILITY_STAT)] = НовоеВремя`.
3. **Агрегация стаков (Stack Aggregation)**: Если бафф стакается (как Cursed Mark), запрещено плодить новые элементы в массивах `active_buffs` для каждого стака. Бафф регистрируется 1 раз (на 1-м стаке). При последующих стаках необходимо лишь обновлять значение существующего слота: `victim.active_buff_values[victim.active_buff_ids.index(BuffId.ABILITY_STAT)] += bonus`.

### 4.2 HUD-записи об эффектах (HUD buffs)

Отображение активных баффов/дебаффов списком в правой части экрана.
- **Добавление HUD-записи**:
  Данные записываются в формате `[TargetPlayer, ExpireTime, NameString, Color, BuffId]`. Затем вызывается подпрограмма `add_hud_buff()`. Пример:
  `eventPlayer.buff_append_args = [victim, getTotalTimeElapsed() + duration, "Mark", Color.PURPLE, BuffId.MARK]`
- **Обновление/Продление времени**:
  HUD-баффы продлеваются путем прямого обновления таймера через встроенный метод поиска индекса:
  ```python
  if BuffId.MARK in victim.hud_buff_ids:
      victim.hud_buff_expires[victim.hud_buff_ids.index(BuffId.MARK)] = НовоеВремяИстечения
  ```
- **Удаление HUD-записи**: Происходит автоматически в правиле `Process HUD buffs` по истечении времени.

## 5. Правила для нейросети

1. **Не менять код без прямого указания** пользователя
2. **Перед изменением массива** — проверь `settings/array_schemas.opy`
3. **Никогда не используй магические числа** для индексов массивов — используй enum
4. **При добавлении нового массива-структуры** — создай enum и member-макросы в `array_schemas.opy`
5. **Всегда добавляй `wait()`** в циклы
6. **Помни лимиты**: 128 playervar, 128 globalvar, 256 эффектов, 128 текстов, 8-12 игроков
7. **Не создавай лишние эффекты** — переиспользуй существующие через reeval
8. **Файл = один логический модуль** — не смешивай разные системы в одном файле
9. **Логирование** — для тестирования и отладки нерабочих механик должна использоваться функция `logToInspector()`
10. **Приоритет инфраструктуры** — для новых механик всегда используй существующую инфраструктуру проекта (без создания локальных "костылей"). Если её недостаточно, универсально дополняй текущую систему с учетом экономии ресурсов сервера, читаемости и масштабируемости.
11. **Обязательная HUD-индикация эффектов** — при наложении любого действия/эффекта на игрока обязательно выводи ему HUD-надпись (бафф/дебафф) справа на экране (по примеру `classes/reader/prayer_of_salvation.opy` или `cursed_mark.opy`).
12. **Атомарные операции с баффами** — при создании и продлении эффектов (баффов/дебаффов) СТРОГО ЗАПРЕЩЕНО использовать локальные циклы `while`. Используйте исключительно атомарные обновления массивов по индексу (подробнее в п. 4.1).

## 6. Архитектура, оптимизация и обход критических уязвимостей

### 6.1 Управление памятью и структурами данных
*   **Избегание «разреженных массивов» (Sparse Arrays):** Категорически запрещено привязывать ID (например, 10000) напрямую к индексу массива. Сервер аллоцирует все промежуточные значения, что ведет к скрытым утечкам памяти и падению сервера (не отображается в Server Load). Массивы должны быть строго плотными и линейными.
*   **Ограничения многомерных массивов:** Модификация элементов на глубоких уровнях вложенности вызывает ресурсоемкую пересборку всего массива.
*   **Отказ от циклов для массивов:** Для перебора, фильтрации и мутации данных инвентаря запрещено использовать классические циклы (while/for). Необходимо всегда использовать нативные функции-мапперы `Mapped Array` (в OverPy - `[x for x in array]`), `Filtered Array` (`[x for x in array if condition]`), `Sorted Array` (выполняются как SIMD-инструкции).

### 6.2 Распределение нагрузки и планировщик (Wait-Yielding)
*   **Прерывания в тяжелых блоках:** При объемной инициализации (спавн боссов, загрузка инвентаря) обязательно использовать `wait(0.016, WaitBehavior.IGNORE_CONDITION)` каждые 5-10 тяжелых действий.
*   **Избегание Continuous Reevaluation:** Запрещено передавать сложные формулы (тригонометрия, поиск по массиву) в параметры HUD или визуальных эффектов. Значения нужно вычислять отдельно по таймеру и передавать статичные переменные.
*   **Делегирование в подпрограммы (Subroutines):** Тяжелую, но редко вызываемую логику нужно выносить в подпрограммы. Они не нагружают планировщик фоновыми проверками условий.

### 6.3 Оптимизация правил (Rules & Conditions)
*   **Сужение контекста событий:** Определять события нужно максимально узко на уровне `@Event` (например, `@Event eachPlayer`, `if eventPlayer.getCurrentHero() == Hero.ANA`), а не через глобальные события с ручными проверками всех игроков.
*   **Отказ от массивов в условиях:** Не использовать сложные проверки с массивами (например, аналог `Array Contains`) внутри условий (`@Condition`). Любое изменение массива инвалидирует кэш и перегружает процессор.
*   **Ленивые вычисления (Short-Circuit):** Самые легковесные и наиболее вероятно ложные условия всегда должны стоять первыми в блоке `AND`.

### 6.4 Паттерн Object Pooling (Пул объектов)
*   **Запрет на динамический спавн в бою:** Строго запрещено постоянно создавать и уничтожать ботов в моменты вычислительной нагрузки. Все боты должны создаваться при инициализации (или с сохранением).

### 6.5 Оптимизация интерфейса (HUD) и работа с текстом
*   **Каскадная конкатенация:** Для экономии лимита в 128 текстов интерфейс должен собираться в массивные блоки с помощью `"{0}{1}{2}".format(A, B, C)`.
*   **Переносы строк:** Эскейп-последовательности `\n` нестабильны. В коде должны использоваться реальные, скопированные символы переноса строки (что дает необходимое выравнивание по левому краю).
*   **Предотвращение Race Condition:** Обязательно вставлять `wait(0.016)` между удалением (`destroyHudText()`) и созданием нового текста.
*   **Сборка мусора (Garbage Collection):** При выходе игрока (событие Player Left Match) скрипт обязан принудительно удалять его локальные HUD-объекты, иначе они остаются в памяти сервера как Ghost Entities.

### 6.6 Обход багов (2024–2026)
*   **Projectile Leak (Фантомные снаряды):** Нативные `createProjectile` навсегда съедают лимит эффектов. Оружие и магия должны симулироваться математическим перемещением обычных эффектов из пула.
*   **Поломка Cooldown API:** Чтение кулдаунов может не работать. Способности, зависящие от талантов, необходимо отключать и симулировать их логику (кулдаун и заряды) через переменные.
*   **Баг масштабирования хитбоксов:** При увеличении размера сущности (рейдовые боссы через `startScalingPlayer`) нативная полоска здоровья улетает за экран. Ее нужно отключать и заменять на кастомный In-World текст/бар.
*   **Баг ИИ ролевых ботов:** Не использовать `heroOf()` на ботах с Role AI для отслеживания текущего героя. Использовать прямую фильтрацию по массиву.

## 7. Настройки лобби (Workshop Settings)

Для корректной работы режима в самом клиенте Overwatch необходимо установить следующие настройки (Settings -> Modes -> All):

- **Максимум игроков**: 8 игроков в лобби (максимум ботов в слоте игрока: 16).
- **Spawn More Dummy Bots**: Выключено на 12 (максимум 4 бота).
- **Beam Effects**: Включены лучевые эффекты и лучевые звуки.
- **Buff/Debuff**: Эффекты статуса баффов/дебаффов включены. Звуки баффов/дебаффов выключены.
- **Explosions**: Энергетические взрывы и звуки взрывов выключены. Кинетические эффекты взрывов включены.
- **Play More Effects**: Включено.
- **Projectiles**: Снаряды (projectiles) включены.
- **Минимальная задержка (Time Delay)**: 0.016 сек.
